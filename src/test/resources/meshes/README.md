# Reference meshes

These are reference triangulations of the shared test scenarios (see
`ScenarioFixtures`), produced by the C Triangle reference and used to prove the
contract validators correct and to drive the phase-by-phase port tests.

There are three files per scenario, one per phase of meshing:

| File | Phase | Triangle switches |
|---|---|---|
| `<name>.delaunay.txt` | Delaunay triangulation of the input **points only** (no segments) | `nzQ` |
| `<name>.cdt.txt` | constrained Delaunay: segments recovered, holes carved, regions attributed, **not refined** | `p[A]nzQ` |
| `<name>.txt` | final mesh: the above **plus quality refinement** | `pq..[Aa]nzeQ` |

Scenarios: `pslg_square`, `pslg_square_quality`, `regions`, `hole`,
`segment_recovery`, `segment_intersection`, `concave_lshape`.

## Format

Plain text, one mesh per file. Six count headers, then sections of indexed
rows. Coordinates are written at full `%.17g` precision so they parse back to
the identical `double`.

```
numberofpoints N
numberofcorners 3
numberoftriangles T
numberoftriangleattributes A
numberofsegments S
numberofedges E
points
<i> <x> <y>                      (N rows)
triangles
<i> <c0> <c1> <c2>               (T rows; corner = point index)
triangleattributes               (only if A > 0, e.g. regions)
<i> <a0> ...
neighbors
<i> <n0> <n1> <n2>               (triangle index opposite corner j, or -1)
segments                         (only with -p)
<i> <p0> <p1>
segmentmarkers                   (only with -p)
<i> <marker>
edges                            (only with -e; the final .txt files only)
<i> <p0> <p1>
```

Sections appear only when the corresponding data exists: the `.delaunay` files
have no `segments`/`segmentmarkers` (no `-p`); only the `regions` meshes have
`triangleattributes`; only the final `.txt` files have `edges` (`-e`).

Neighbour convention: `neighbors` row `i` lists the triangle across the edge
**opposite corner `j`** of triangle `i`, or `-1` on a boundary.

## How they are validated — structurally, not byte-for-byte

Unlike `contract/predicates.txt` (whose exact signs a port must reproduce),
these meshes are **not** matched byte-for-byte. A correct reimplementation
produces a *different but equally valid* mesh (different vertex numbering,
triangle order, Steiner-point placement). So:

- `PhaseValidationTest` loads each `.delaunay`/`.cdt` mesh and checks it against
  the contract invariants appropriate to its phase (via `MeshValidator`),
  proving the validator is a correct phase-by-phase oracle.
- `ContractTest` does the same for the final `.txt` meshes.
- The port tests (`DelaunayTriangulatorTest`, `ConstrainedDelaunayTriangulatorTest`,
  `JavaTriangleMesherTest`) validate the **Java mesher's own output** against the
  same contract — they do not compare to these files.

## Provenance

Generated by the reduced C Triangle reference repository:

- `<name>.txt` — `test/golden_runner.c`
- `<name>.delaunay.txt`, `<name>.cdt.txt` — `test/phase_runner.c`

Both read the shared `test/scenarios.c`. Regenerate there and re-copy if
`triangle.c` changes. Do not edit by hand.
